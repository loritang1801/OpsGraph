# OpsGraph PRD

- 版本：v0.1
- 日期：2026-03-16
- 产品定位：面向中小技术团队的 `Incident Response` Multi-Agent 平台

## 1. 产品摘要

OpsGraph 是一个以事故响应为核心闭环的多 Agent 系统，面向 `SRE`、值班工程师和技术负责人，帮助团队从告警接入、上下文聚合、假设生成、runbook 建议、事故沟通到 `RCA / postmortem` 沉淀形成一条可回放、可评估、可优化的响应链路。

v1 目标不是替代值班工程师，也不是构建完整的 AIOps 平台，而是把事故响应中最耗时的“信息收集、背景理解、候选根因整理、沟通文案整理”自动化，让人把时间花在判断和处置上。

## 2. 背景与问题定义

中小技术团队做事故响应时通常存在以下问题：

1. 告警来自多个系统，重复告警和噪声会干扰判断。
2. 值班工程师需要手工收集最近变更、服务 owner、历史事故和相关 runbook。
3. 事故沟通高度依赖现场发挥，对内同步和对外通报容易遗漏关键事实。
4. 事故结束后 `RCA` 和 `postmortem` 编写费时，很多经验没有沉淀成可复用记忆。
5. 团队想引入 Agent，但担心自动操作风险和系统不透明。

## 3. 目标与成功标准

### 3.1 业务目标

1. 缩短事故初始响应时间
2. 提高 incident commander 和值班工程师的信息获取效率
3. 把历史事故和 runbook 经验沉淀成可检索资产

### 3.2 产品目标

1. 自动接收并聚合告警，创建统一 incident workspace
2. 自动补全最近部署、服务依赖、owner、历史事故和工单上下文
3. 生成候选根因假设与优先调查路径
4. 生成人工确认后的沟通草稿与复盘草稿
5. 通过 replay、评估和反馈提升后续事故处理质量

### 3.3 北极星指标

1. `Time to first actionable context`
2. `Hypothesis top-k hit rate`
3. `Comms draft adoption rate`
4. `Replay pass rate`
5. `Mean time to acknowledge` 的改善幅度

### 3.4 v1 验收阈值

1. 高优事故在 `2` 分钟内生成首版上下文摘要
2. 历史事故回放中，候选根因 top-3 命中率达到 `>= 60%`
3. 沟通草稿直接采用或少量编辑后采用比例达到 `>= 70%`
4. 所有调查结论必须附带证据来源

## 4. 非目标

v1 不做以下能力：

1. 不做无人值守的自动修复或自动回滚
2. 不替代监控、日志和 APM 平台本身
3. 不覆盖容量规划、成本优化等广义运维问题
4. 不实现复杂企业级排班、审批和多区域容灾编排
5. 不在没有人工确认的前提下执行高风险写操作

## 5. 目标用户

### 5.1 核心用户

1. `On-call Engineer`
   负责第一时间接警并判断是否需要升级。
2. `SRE / 平台工程师`
   负责分析系统上下文、执行 runbook 和推动恢复。
3. `Tech Lead / Engineering Manager`
   负责事故协调、跨团队沟通和复盘质量。

### 5.2 次级用户

1. `Incident Commander`
   关注时间线、关键判断和沟通节奏。
2. `产品/客服负责人`
   主要消费事故状态更新和影响范围说明。

## 6. 核心使用场景

### 场景 1：接收高优告警

系统接到 `Prometheus` / `Grafana` 告警 webhook，自动识别是否属于新事故或已有事故的新增信号。

### 场景 2：自动补全上下文

系统查询最近 `GitHub` 部署记录、相关 `Jira` 变更单、服务 owner、历史事故与 runbook，并在 incident workspace 中集中展示。

### 场景 3：生成调查假设

系统基于告警模式、服务依赖和历史相似事故，生成候选根因及建议调查顺序。

### 场景 4：协助事故沟通

系统根据当前已确认事实生成对内同步和对外通报草稿，等待人工确认后发出。

### 场景 5：事故结束后沉淀经验

系统自动整理时间线、关键动作、最终根因和后续改进项，生成 `postmortem` 草稿。

## 7. Jobs To Be Done

1. 当我接到告警时，我希望快速看到这次事故最相关的上下文，这样我不用手工切换多个系统。
2. 当我要判断优先调查什么时，我希望系统给出可信的候选根因和证据，这样我能更快进入有效排查。
3. 当我需要同步事故状态时，我希望系统自动整理沟通草稿，这样我可以减少低价值复制工作。
4. 当事故结束后，我希望系统能自动沉淀时间线和经验，这样后续团队能真正复用。

## 8. v1 范围

### 8.1 范围内

1. 只读接入 `Prometheus / Grafana` 告警
2. 读取 `GitHub` 最近部署与提交元数据
3. 读取 `Jira` issue / deploy ticket
4. 输出 `Slack` 或 `飞书` 沟通草稿
5. 建立 incident workspace、时间线、候选根因、runbook 建议、复盘草稿
6. 历史事故 replay 与评估
7. 服务级和团队级 memory

### 8.2 范围外

1. 自动执行生产变更
2. 自动扩缩容或自动修复
3. 全量日志检索平台替代
4. 成本、性能、容量等非事故主线能力

## 9. 关键流程

### 9.1 端到端流程

1. 接收告警 webhook
2. `triage agent` 完成去重、聚合、初步定级
3. 系统创建或更新 incident workspace
4. `investigator agent` 拉取变更、owner、历史事故和依赖信息
5. `hypothesis engine` 生成候选根因与调查建议
6. `runbook advisor` 推荐处置路径
7. `comms agent` 生成事故沟通草稿
8. 工程师确认处置与对外同步
9. 事故结束后 `postmortem reviewer` 生成复盘草稿并沉淀 memory

### 9.2 人工介入点

1. 事故定级确认
2. runbook 执行前确认
3. 外部沟通发送前确认
4. 最终根因与改进项确认

## 10. 功能需求

### FR-1 告警接入与去重

系统必须支持：

1. 接收 `Prometheus / Grafana` 告警 webhook
2. 根据告警标签、服务名、时间窗口和特征做去重聚合
3. 判断属于新事故还是已有事故更新

### FR-2 Incident Workspace

每个 incident 必须有统一工作区，至少包含：

1. 基础信息：标题、严重度、状态、开始时间
2. 信号列表
3. 已确认事实
4. 候选根因
5. runbook 建议
6. 沟通草稿
7. 时间线

### FR-3 上下文聚合

系统必须自动拉取并展示：

1. 最近部署和提交信息
2. 相关 `Jira` 变更单
3. 服务 owner 与升级路径
4. 历史相似事故
5. 相关 runbook

### FR-4 候选根因生成

系统必须输出：

1. top-k 候选根因
2. 每条假设的证据依据
3. 建议的验证步骤
4. 假设间的优先级排序

### FR-5 Runbook 建议

系统必须根据 incident 类型推荐 runbook，并区分：

1. 只读调查动作
2. 低风险建议动作
3. 高风险动作

高风险动作在 v1 中只允许展示建议，不允许自动执行。

### FR-6 事故沟通草稿

系统必须能生成：

1. 对内状态同步草稿
2. 对外状态页草稿
3. 事故结束通知草稿

每份草稿都必须基于已确认事实，不允许凭空扩写。

### FR-7 时间线与复盘

系统必须自动整理：

1. 告警触发时间
2. 关键判断与处置动作
3. 恢复时间
4. 初步根因
5. 改进项与责任人建议

### FR-8 Memory

系统必须维护：

1. `服务级记忆`
   包括依赖关系、常见故障模式、已知 runbook、owner。
2. `团队级记忆`
   包括历史事故、复盘结论、常用沟通模板、升级习惯。
3. `incident 级短期记忆`
   包括当前上下文、确认事实、未决问题和处理中动作。

## 11. Agent 设计要求

v1 采用显式多 Agent 协作。

1. `triage agent`
   负责告警聚合、定级建议和去重。
2. `investigator agent`
   负责上下文拉取、事件关联和证据整理。
3. `runbook advisor`
   负责推荐调查步骤与处置建议。
4. `comms agent`
   负责对内外沟通草稿。
5. `postmortem reviewer`
   负责时间线整理、复盘草稿和经验沉淀。

## 12. LangGraph 状态流要求

v1 必须采用状态机编排。建议最小状态集如下：

1. `detect`
2. `triage`
3. `enrich`
4. `hypothesize`
5. `advise`
6. `communicate`
7. `resolve`
8. `retrospective`

状态约束：

1. `triage` 后才能进入 `enrich`
2. `communicate` 只能引用已确认事实
3. `resolve` 后必须进入 `retrospective`
4. 所有高风险动作必须卡在人审节点

## 13. Skills / 工具集成要求

v1 默认以读取和建议为主。

1. 告警 webhook 接收器
2. `Prometheus / Grafana` 告警读取
3. `GitHub` 部署与提交元数据读取
4. `Jira` 读取
5. `Slack / 飞书` 消息草稿生成
6. 向量检索与 incident replay 查询

禁止行为：

1. 无人审自动回滚
2. 无人审自动发对外公告
3. 无人审自动执行可能影响生产环境的命令

## 14. 数据与对象模型

PRD 级别需固定以下核心对象：

1. `Incident`
2. `Signal`
3. `Service`
4. `DeploymentEvent`
5. `Hypothesis`
6. `RunbookRecommendation`
7. `TimelineEvent`
8. `CommsDraft`
9. `Postmortem`
10. `MemoryRecord`

关键字段要求：

1. 每个 `Hypothesis` 必须包含证据链接和验证步骤
2. 每个 `TimelineEvent` 必须记录来源和时间戳
3. 每个 `CommsDraft` 必须记录基于哪些已确认事实生成

## 15. 评估、测试与优化要求

### 15.1 离线评估

至少建设以下评估集：

1. 历史事故 replay 集
2. 告警去重与分组数据集
3. 根因排序标注集
4. 沟通草稿忠实度样本集

### 15.2 测试策略

1. 单元测试：告警归并、状态迁移、模板渲染、引用校验
2. 工作流测试：模拟一次完整事故从检测到复盘
3. 回归测试：固定历史事故回放结果
4. 安全测试：验证任何高风险动作都需要人工确认

### 15.3 在线评估

1. 首版上下文产出时延
2. 根因建议采纳率
3. 沟通草稿采用率
4. replay 结果稳定性
5. 工程师反馈标签分布

### 15.4 优化路径

v1 优先做：

1. 事件聚合策略优化
2. 假设排序优化
3. 历史事故检索优化
4. 状态流与重试策略优化

微调仅用于：

1. 告警分类
2. 事故类型识别
3. 根因候选排序

v1 不微调主推理模型，不做端到端自治运维。

## 16. 非功能需求

1. `透明性`：所有假设和建议必须可追溯到证据
2. `安全性`：高风险动作必须人工确认
3. `稳定性`：单一外部系统不可用时，incident workspace 仍可用
4. `时效性`：高优事故上下文补全要足够快，v1 目标为 `2` 分钟内产出首版摘要
5. `可回放性`：每次 incident 轨迹都应可用于后续 replay 评估

## 17. 风险与约束

1. 外部监控和项目管理系统的数据质量会影响建议质量
2. 历史事故结构化程度不足时，replay 和 memory 效果有限
3. 如果沟通草稿和事实不一致，用户会迅速失去信任
4. 若状态流设计过于自由，会导致 incident 过程不可控且难评估

## 18. 商业化与包装假设

v1 适合的初始客户画像：

1. `10-100` 人规模的 SaaS / 平台型技术团队
2. 已有基础监控和 issue 流程，但事故响应仍靠人工协调的团队
3. 想引入 Agent，但对生产环境自动化有顾虑的团队

建议收费逻辑：

1. 按服务数或 incident 数收费
2. 历史 replay、复盘知识库和高级集成作为高阶套餐

## 20. 验收标准

需求文档层面的完成定义：

1. 明确 `OpsGraph` 不是通用 AIOps 平台，而是事故响应闭环产品
2. 明确谁在什么时候使用系统，以及人工接管点在哪里
3. 明确 Agent、memory、LangGraph、eval、testing 与优化的职责边界
4. 可以直接进入架构设计，而不需要再补关键产品决策
